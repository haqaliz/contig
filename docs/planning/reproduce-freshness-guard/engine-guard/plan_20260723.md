## 0. Correction to the PRD's effort estimate (read first)

**PRD M7 says ~42 tests need `run_started_at` threaded. That number is wrong, and it was
wrong in a way that makes the slice cheaper, not more expensive.**

It assumed the guard would be compared against a *realistic* run start. It will not be. The
shipped notebook tests use a **fixed past epoch**, `_RUN_START = 1_000_000.0`
(`test_reproduce.py:2343`) — a 1970-era timestamp. Any file a test writes during execution has
a real mtime (~1.7e9), which is **enormously greater** than `_RUN_START`. Therefore:

> Defaulting `run_started_at=_RUN_START` in the two shared `_run` test helpers makes every
> existing located and env-resurrection test **pass unchanged**, because their fixtures are
> written at real wall-clock time and are trivially "fresh" relative to a 1970 stamp.

The 15 status-flips and 6 message-assertion breaks the dig predicted **do not occur** under
this design. Revised, verified churn:

| Edit | Count | Why |
|---|---|---|
| `_run` helper defaults (`test_reproduce.py:1255`, `test_reproduce_env_resurrection.py:105`) | 2 | Covers ~60 located + env tests at once |
| Direct `run_reproduction(...)` calls in the flat section (`test_reproduce.py:1111, 1129, 1145, 1162, 1180, 1200, 1216, 1232`) | 8 | No helper exists there; each needs an explicit stamp |
| `test_run_reproduction_notebook_missing_run_started_at_raises` (`:2490`) | 1 | Must now pass `run_started_at=None` explicitly to defeat the new default |
| **Total existing-test edits** | **11** | vs. the ~42 estimated |
| New tests | ~12 | Headline stale ×4, raise ×4, stdout-exempt, retry, symlink, fresh-classification |

This is a plan-level correction to the PRD, recorded here rather than silently applied.

## 2. Design decisions this plan settles

### D1 — Helper shape and placement

A nested function inside `run_reproduction` (it closes over `run_started_at`), defined
immediately before `_observe_located`:

```python
def _require_fresh(resolved: Path, noun: str, label: str) -> str | None:
    """Return an UNVERIFIED message when `resolved` predates this run, else None.

    The guard is non-bypassable: an unstamped run_started_at is a programming
    error (a None meaning "guard off" would silently disable a false-pass
    guard), so it raises rather than degrading.

    A file that cannot be stat()'d returns None -- NOT a freshness failure.
    The caller's own missing/unreadable path owns that message, so this helper
    never pre-empts a more specific existing error.
    """
    if run_started_at is None:
        raise ValueError("run_started_at is required to verify a located claim")
    try:
        mtime = resolved.stat().st_mtime
    except OSError:
        return None
    if mtime < run_started_at:
        return f"{noun} {label} was not rewritten by this run (mtime predates run start)"
    return None
```

**Why `OSError → None`:** returning a "missing" message here would pre-empt the four existing
missing-file messages and cause exactly the R3 churn we can otherwise avoid. Delegating keeps
every current error path intact.

### D2 — Placement within each observer: after containment, before the cache read

All three observers share an identical prologue. The guard goes immediately after the
`relative_to(repo_root)` containment check and **before** the `key = str(resolved)` cache
lookup, so a stale file is never parsed and its parse never enters the cache.

**Deliberate divergence from PRD M3's literal wording.** M3 said "size check before mtime,
preserving `:1094`-then-`:1100`". That ordering is internal to the notebook branch's single
`stat()`. For the pattern observer the size check lives *inside* the cache-miss block
(`:1009-1022`), which the guard must precede. So for a file that is both stale and oversized,
staleness is reported. No test covers that combination, and M3's primary clause ("a stale file
is never parsed") is what actually matters. The oversized fixture stays fresh (Phase 0), so
`test_run_reproduction_pattern_claim_oversized_file_is_unverified_and_not_read` (`:2230`)
still exercises the size path and still asserts its byte count.

### D3 — R4, symlinks: follow them (keep `stat()` default), and pin it with a test

`Path.resolve()` already resolves symlinks, and `stat()` follows them. **Keep that.** A
symlink pointing at a fresh target *is* fresh; a symlink pointing into a stale shared cache is
correctly `UNVERIFIED`, because this run did not produce that content. `follow_symlinks=False`
would be wrong — it would stat the link inode, which a `ln -s` during the run makes fresh even
when the payload is ancient, converting a real staleness into a false pass. Add a test pinning
both directions.

### D4 — R5, clock skew: no code, disclosure only

Same accepted posture as R1: no tolerance. Recorded in the CHANGELOG's honest-limits paragraph.

### D5 — Flat results: a stale file must not claim to be unparseable

At `:1226-1234` the read becomes:

```python
results_file = repo_path / results_path
results: dict | None = None
results_stale: str | None = None
if results_file.exists():
    results_stale = _require_fresh(results_file, "results file", repr(results_path))
    if results_stale is None:
        ... existing parse, unchanged ...
```

and in the flat branch, `results_stale` is checked **before** the existing
`if results is None:` arm, emitting the freshness message. A stale-but-valid results file is
never described as "missing or unparseable" (PRD M5).

## 5. Testing Strategy

- **Unit:** per phase above. Every new test is deterministic — fixture files with
  `os.utime`-stamped mtimes against the fixed `_RUN_START` epoch, injected `run_started_at`,
  scripted executors. **No real repo, network, notebook execution, or pip.**
- **Integration:** `tests/test_cli_reproduce.py`, unmodified, as the "real runs still work"
  proof.
- **Full suite:** `uv run pytest` before the final commit.
- **No dashboard work** in this aspect.

## 6. Edge Cases & Error Handling

| Case | Behavior |
|---|---|
| File missing | Existing missing message (helper returns `None` on `OSError`) |
| File stale | `UNVERIFIED`, "was not rewritten by this run (mtime predates run start)" |
| Stale **and** oversized | Staleness reported (D2), no test asserts otherwise |
| Stale **and** unparseable | Staleness reported; never parsed |
| Symlink → fresh target | Resolves (D3) |
| Symlink → stale target | `UNVERIFIED` (D3) |
| stdout-mode pattern | **No guard, ever** |
| `run_started_at is None` on a guarded branch | `ValueError`, loud |
| `--allow-install` retry rewrites the file | Resolves — stamp taken once, before the first run |
| Coarse mtime / clock skew | False `UNVERIFIED`, accepted (R1/R5), no tolerance |

## 7. Agent Execution Notes

- **Checkpoint after Phase 0.** If the suite is not fully green with zero behavior change,
  stop — the stamp design is wrong and everything after it is unsound.
- **Hard checkpoint at Phase 5.** If `tests/test_cli_reproduce.py` requires *any* edit,
  **stop and report**. That is R2 materializing, and it is a product decision (the PRD
  committed to strict-with-no-opt-out), not something to fix in code.
- **Do not add a tolerance/fudge window** to the mtime comparison under any circumstance
  (R1). If a test seems to need one, the test is wrong.
- **Do not guard the stdout-mode pattern arm.** It is immune by construction; guarding it
  would break a shipped, correct behavior.
- **Ask before** changing `models.py` or adding a structured staleness field — the PRD
  deferred that (S3) in favor of a greppable message stem.
- Phases 1-4 are near-identical in shape and can be executed by one agent each, but they all
  touch the same two files, so run them **sequentially**, not in parallel, to avoid conflicts.
