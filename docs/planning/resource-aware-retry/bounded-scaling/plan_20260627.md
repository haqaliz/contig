## Impact analysis (current state, grounded)

- `apply_patch` (`self_heal.py:259-300`) scales `memory`/`time` with no cap:
  `int(_lead_number(limits.get(k), default) * factor)` → `"{n}.GB"` / `"{n}.h"`.
  Defaults: `_DEFAULT_MEMORY_GB = 8`, `_DEFAULT_TIME_HOURS = 4` (`self_heal.py:39-40`).
- Loop safe-patch tail (`self_heal.py` end of `while True`): `if attempt >=
  max_attempts: gave_up; else apply_patch(safe); record patched_and_retried; attempt
  += 1`. This is the insertion point for the at-ceiling give-up — **before**
  `apply_patch`, **after** the `max_attempts` block.
- `propose_patches` (`repair.py:16-35`) emits the resource patch for `oom`
  (`{"multiply":{"memory":2}}`) and `time_limit` (`{"multiply":{"time":2}}`).
  **Unchanged** — the policy lives where numbers are applied.
- `RepairStep` (`models.py:225-231`): `attempt:int, diagnosis, patch:Patch|None,
  outcome:str`. No message field → add `detail: str | None = None`.
- Failed attempts already stash to the pending corpus (`self_heal.py:364-377`,
  `append_case`) and stream to `repair_progress.jsonl` (`_record_attempt`,
  `self_heal.py:77-90`). Reused as-is.

## Phase 1 — Ceiling clamp + never-shrink in `apply_patch`

**Goal:** `apply_patch` never emits a resource value above the cap, and never
shrinks a pre-existing request already at/above the cap.

**Prerequisites:** none.

**Files:**
- modify `src/contig/self_heal.py` (constants + `apply_patch` signature/body)
- modify `tests/test_self_heal.py` (new clamp/never-shrink tests) — confirm the
  exact test module during RED (`grep -rl "apply_patch" tests/`).

**RED — write these tests first:**
1. `test_apply_patch_clamps_memory_to_ceiling`: target `resource_limits={"memory":
   "96.GB"}`, OOM patch `{"multiply":{"memory":2}}`, `ceiling={"memory":128,...}` →
   result memory == `"128.GB"` (not `"192.GB"`).
2. `test_apply_patch_clamps_time_to_ceiling`: target time `"48.h"`,
   `{"multiply":{"time":2}}`, cap 72 → `"72.h"` (not `"96.h"`).
3. `test_apply_patch_does_not_shrink_oversized_request`: memory `"256.GB"`, OOM
   multiply 2, cap 128 → memory stays `"256.GB"` (never reduced).
4. `test_apply_patch_scales_normally_below_ceiling`: memory `"8.GB"`, multiply 2,
   cap 128 → `"16.GB"` (regression: unchanged behaviour well below the cap).

**GREEN — implementation steps:**
1. Add `CEILING_MEMORY_GB` / `CEILING_TIME_H` constants.
2. Change signature to `apply_patch(target, patch, params=None, *, ceiling: dict[str,
   int] | None = None)`; default `ceiling = {"memory": CEILING_MEMORY_GB, "time":
   CEILING_TIME_H}` when None.
3. In the `resource` branch, for each of `memory`/`time`:
   - `current = _lead_number(limits.get(k), default)`
   - `bumped = int(current * int(mult[k]))`
   - `capped = min(bumped, ceiling[k])`
   - `final = max(capped, int(current))`  # never shrink a pre-existing larger value
   - write `f"{final}.GB"` / `f"{final}.h"`.
4. Leave all non-resource branches untouched.

**Validation:** `uv run pytest tests/test_self_heal.py -k "apply_patch" -q` green;
full `uv run pytest` still green.

**Commit:** `feat(self-heal): clamp auto-scaled resources to an absolute ceiling`

## Phase 2 — `RepairStep.detail` + at-ceiling give-up in the loop

**Goal:** when a resource failure recurs and the relevant resource is already
at/above its cap, the loop stops with a distinct `gave_up_at_ceiling` outcome + a
human `detail` message, returns the FAILed `RunRecord`, and the case is captured —
never a false PASS, never an infinite loop.

**Prerequisites:** Phase 1.

**Files:**
- modify `src/contig/models.py` (`RepairStep.detail`)
- modify `src/contig/self_heal.py` (ceiling param on `self_heal_run`; helper
  `_resource_ceiling_block`; insert give-up branch)
- modify `tests/test_self_heal.py` (give-up / termination / recovery / corpus tests)

**RED — write these tests first** (use a fake `Executor` that returns exit 137 and
writes an OOM log; for recovery, succeed on the 2nd call):
1. `test_gives_up_at_memory_ceiling`: target memory at the cap (`"128.GB"`),
   persistent OOM → the final `RepairStep.outcome == "gave_up_at_ceiling"`,
   `.detail` contains "memory" and "128"; a `RunRecord` is returned; verdict is FAIL
   (assert it is **not** PASS/UNVERIFIED).
2. `test_gives_up_at_time_ceiling`: time at `"72.h"`, persistent `time_limit`
   (fake writes a "time limit" log) → `gave_up_at_ceiling`, detail names time/72.
3. `test_persistent_oom_terminates`: persistent OOM, low starting memory, `ceiling`
   small enough to be reached within `max_attempts` (e.g. cap 16, start 8, multiply
   2) → call returns (no hang), attempts bounded, ends `gave_up_at_ceiling`.
4. `test_oom_recovers_below_ceiling_unchanged`: OOM then success, well below the cap
   → ends recovered (verdict not FAIL), last resource step `patched_and_retried`
   (regression that Phase 2 didn't break recovery).
5. `test_ceiling_giveup_is_captured_in_pending_corpus`: assert the gave-up run
   appended a case to the provided `pending_corpus` path.

**GREEN — implementation steps:**
1. `models.py`: add `detail: str | None = None` to `RepairStep` (backward
   compatible; appears in `model_dump_json`).
2. `self_heal.py`: add `resource_ceiling: dict[str, int] | None = None` to
   `self_heal_run`; resolve to the default dict once at the top of the function.
3. Add helper:
   ```python
   def _resource_ceiling_block(diagnosis, target, ceiling) -> str | None:
       """Message if the resource this failure class scales is already at/above its
       cap (so auto-scaling can grow it no further); else None."""
       if diagnosis.failure_class == "oom":
           if _lead_number(target.resource_limits.get("memory"), _DEFAULT_MEMORY_GB) >= ceiling["memory"]:
               return f"Out of memory persists at the {ceiling['memory']} GB ceiling; needs a node with more memory."
       if diagnosis.failure_class == "time_limit":
           if _lead_number(target.resource_limits.get("time"), _DEFAULT_TIME_HOURS) >= ceiling["time"]:
               return f"Time limit persists at the {ceiling['time']} h ceiling; needs a longer walltime allowance."
       return None
   ```
4. In the safe-patch tail, **after** the `if attempt >= max_attempts` block and
   **before** `apply_patch(current_target, safe, ...)`, insert:
   ```python
   block = _resource_ceiling_block(diagnosis, current_target, resource_ceiling)
   if block is not None:
       _record_attempt(
           run_dir, repair_history,
           RepairStep(attempt=attempt, diagnosis=diagnosis, patch=safe,
                      outcome="gave_up_at_ceiling", detail=block),
       )
       return _finalize(
           exc.record, repair_history, run_dir,
           runs_dir=runs_dir, run_id=run_id, webhook=notify_webhook,
       )
   ```
5. Pass `ceiling=resource_ceiling` into the safe-patch `apply_patch` call (and the
   gated/auto-approve/chosen `apply_patch` calls, for consistency).

**Validation:** `uv run pytest tests/test_self_heal.py -q` green; full `uv run
pytest` green.

**Commit:** `feat(self-heal): give up honestly when a resource is at its ceiling`

## Environment & reproducibility

- `uv sync` already done in the worktree. No new env vars or external tools.
- Determinism preserved: scaling is purely arithmetic on config literals; no trace

  give-up reason are recorded in `repair_history` (in the bundle), strengthening the
  auditable trail.

## Edge cases & error handling

- **Pre-existing request above the cap** (R3): give-up fires (resource ≥ cap), limit
  not shrunk. Covered by `test_gives_up_at_memory_ceiling` variants /
  never-shrink test.
- **Unparseable resource literal:** `_lead_number` already falls back to the default;
  clamp/compare still well-defined.
- **`time_limit` with no time limit set:** falls back to `_DEFAULT_TIME_HOURS = 4`,
  well below 72 → scales normally; correct.
- **Non-resource failure classes:** `_resource_ceiling_block` returns None → no
  behaviour change. Asserted indirectly by the full-suite regression.
- **Both memory and time present in a multiply:** each clamped independently; the
  give-up trigger is keyed on the failure class's own resource only.
