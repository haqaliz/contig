## Phase 2 — the deterministic matcher (RED→GREEN→REFACTOR)

- **Goal:** `extract_claims` finds named-metric + number claims per the spec.
- **Steps (test-first, one behavior per RED):**
  1. **Metric vocabulary constant** `_METRIC_VOCAB` (the spec's enumerated seed) + a slug helper
     `_slug(metric) -> str` (lowercase, non-alnum→`_`, collapse repeats, strip). RED: slug tests
     (`"log2 fold change"` → `"log2_fold_change"`, `"R²"` → `"r"` or `"r2"` — pin the exact rule).
  2. **Number regex** `_NUMBER = re.compile(r"-?\d+(?:\.\d+)?(?:[eE][-+]?\d+)?%?")`. RED: parse
     helper `_parse_number(tok) -> tuple[float|None, str|None]` returns `(value, unit)` with
     `unit="%"` when a trailing `%` present, value = bare number; non-finite → `(None, None)`.
  3. **Match assembly:** build a regex (or scan) that finds a vocab metric word, an optional
     connective/filler within a bounded window (≤ ~40 chars / ≤ ~4 tokens), then a number.
     Connectives: `of|was|is|=|:|reached|achieved|at`. RED: the canonical positives
     (`AUC of 0.91`, `accuracy of 87%`, `F1 = 0.83`, `log2 fold change of -2.3`,
     `Pearson correlation of 0.76`) each yield one claim with the right `metric`, `value`, `unit`.
  4. **Inequality skip:** if the char(s) immediately before the number (post-trim) are
     `<`,`>`,`≤`,`≥`,`<=`,`>=` → skip. RED: `p < 0.001` and `accuracy > 0.9` yield **no** claim.
  5. **source_text:** capture the sentence/snippet around the match (split on `.`/newline, or a
     ±N-char window). RED: `source_text` contains the metric and the number.
  6. **id generation + uniquify:** ids = `_slug(metric)` uniquified in first-appearance order
     (`auc`, `auc_2`, …). RED: two AUC values → ids `auc`, `auc_2`, deterministic across runs.
  7. **De-dup:** key `(slug, value)`, file-wide, keep first `source_text`. RED: same metric+value
     twice → one claim; distinct metrics same value → two claims.
  8. REFACTOR: factor the scanner cleanly; keep every branch total (never raises).
- **Validation:** `uv run pytest tests/test_claim_extraction.py -q` green.
- **Commit:** `feat(claim-extraction): deterministic named-metric number extractor`

## Phase 3 — robustness + the labeled fixture corpus (RED→GREEN)

- **Goal:** the "wild inputs never raise" guarantee + the measurable-recall bar.
- **Steps:**
  1. RED: a `test_extract_claims_never_raises_on_wild_inputs` (mirror
     `test_reproduce_locator.py:118-143`) over `""`, non-str (pass `None`/`123` guarded),
     control chars, a 100-number prose blob → always a list, never an exception.
  2. RED: **labeled fixture corpus** — 3–5 short `.md`/`.txt` excerpt strings, each with a
     hand-labeled expected `set[(slug, value, unit)]`. Assert `extract_claims` recovers **every**
     labeled claim and emits **zero malformed** entries (finite float value, non-empty id).
     Include a percentage case, an inequality case (absent from expected), and a duplicate case.
  3. GREEN: fix any misses/over-matches the corpus reveals by adjusting the window/vocab
     (conservative — prefer precision).
- **Validation:** full `uv run pytest tests/test_claim_extraction.py -q` green; then `uv run
  pytest -q` (whole suite, confirm zero blast radius).
- **Commit:** `test(claim-extraction): labeled fixture corpus + wild-input robustness`

## Edge cases

- `R²`/`r-squared` slug collisions with bare `r` — vocabulary uses explicit multiword forms;
  document the chosen slug for each.
- Scientific notation (`1.2e-3`) and negative values (`-2.3`) must parse.
- A number with a trailing `%` inside an inequality (`> 90%`) → skipped (inequality wins).
- Windows too greedy → prose noise; keep tight, let the corpus tune it.
