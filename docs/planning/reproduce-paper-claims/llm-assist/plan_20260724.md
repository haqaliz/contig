- **Steps (test-first; `tests/test_claim_extraction_llm.py`, mirroring `test_detect_llm.py`):**
  1. RED: with no `CONTIG_LLM_PROVIDER` env → `extract_with_llm("...")` returns `[]` and the seam

     the lazy-import seam (copy `detect._llm_complete`'s shape; SDK imported **inside**).
- **Validation:** `uv run pytest tests/test_claim_extraction_llm.py -q`.
- **Commit:** `feat(claim-extraction): optional env-gated LLM extraction seam`

## Phase 2 — defensive reply parse (RED→GREEN)

- **Goal:** `_claims_from_reply(reply) -> list[ExtractedClaim]` — tolerant and never-raising.
- **Steps (test-first):**
  1. RED: a clean JSON list reply → `ExtractedClaim`s with `origin="llm"`, ids via the shared
     `_slug` uniquifier, `unit` carried, `value` finite-float-validated (non-bool).
  2. RED: **prose-wrapped** JSON list (`"Here are the claims: [ ... ] hope this helps"`) → parsed
     (list-analogue of `_extract_json_object`; find the first balanced `[...]`).
  3. RED: seam raises / returns `"not json"` / returns a JSON object not a list / an entry missing
     `value` / a non-finite value → each contributes **nothing**, returns `[]`, never raises.
  4. RED: **no API key leak** — set a distinctive dummy key, run, assert it appears in no returned
     `source_text`/id and (if the impl logs) no log (mirror
     `test_detector_does_not_log_the_api_key`).
  5. GREEN: implement, swallowing every exception into `[]` (mirror `_diagnosis_from_reply`).
- **Validation:** `uv run pytest tests/test_claim_extraction_llm.py -q`.
- **Commit:** `feat(claim-extraction): defensive LLM reply parser (never raises)`

## Testing strategy & reproducibility

- **No real provider in CI, ever** — the seam is monkeypatched in every test; the real path is
  shape-asserted only. Env set with dummy values.
- Manual gate (pre-merge, human-run): real provider once; record in PR.

## Edge cases

- Provider configured but SDK not installed → the lazy import inside `_llm_complete` raises; the
  caller swallows → `[]` (degrade to core-only). Add a test with the import monkeypatched to raise.
- Reply with duplicate claims → deduped by the same `(slug, value)` key.
- Circular import risk: `claim_extraction` importing `detect` — `detect` does not import
  `claim_extraction`, so it's safe; confirm no import cycle in a test.

## Manual pre-merge gate (human-run, once)

CI never calls a real provider — the seam is monkeypatched everywhere and the
real `_llm_complete` is shape-asserted only (injecting a fake `anthropic`/`openai`
module into `sys.modules`, asserting the built request's `model`/`max_tokens`/
`messages` and that the key never enters the request, with no network). Before
merging, a human runs the real path **once** to confirm the wiring end-to-end:

```bash
export CONTIG_LLM_PROVIDER=claude   # or openai
export ANTHROPIC_API_KEY=…          # or OPENAI_API_KEY (real key, local only)
uv run python -c "from contig.verification.claim_extraction import extract_with_llm; \
  print(extract_with_llm(open('fixture_paper.txt').read()))"
```

Confirm: claims come back as `ExtractedClaim`s with `origin=='llm'`; the run
never raises; and the key appears nowhere in the output. **Record the result in
the PR description.** This gate is manual by design — a real key must never live
in CI.
