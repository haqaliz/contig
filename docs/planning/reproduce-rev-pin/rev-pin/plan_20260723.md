## 2. Implementation phases

### Phase 1 — Pure `--rev` validator

- **Goal:** a pure predicate that accepts a full SHA / tag / branch and refuses everything
  dangerous or unfetchable, with no I/O. Satisfies spec criteria 1–2.
- **Prerequisites:** none. Parallelizable with Phase 2.
- **Files:** `src/contig/fetch.py` (add), `tests/test_reproduce_rev_pin.py` (new).
- **Steps:**
  1. **RED:** new test file, section header comment in house style. Table-driven accept cases
     (`"a"*40`, `"v2.1"`, `"main"`, `"feature/x"`, `"release-1.0"`) and refuse cases, each
     asserting the refusal *names its cause*.
  2. Add `RevArgument` frozen dataclass mirroring `RepoArgument`/`FetchResult`
     (`fetch.py:21-60`, `:150-181`): `rev: str | None`, `refusal: str | None`, mutual
     exclusion enforced in `__post_init__`, `.accept(rev)` / `.refuse(reason)` classmethods.
  3. Add `classify_rev_argument(arg: str) -> RevArgument`, documenting the decision order in
     the docstring exactly as `classify_repo_argument` does (`fetch.py:102-120`):
     1. **leading `-` refused first and unconditionally** (RCE shape — an option reaching
        `git fetch` in the ref position). Same reasoning, same message shape as `:121-125`.
     2. empty or whitespace-only.
     3. any whitespace or control character anywhere.
     4. **7-to-39 hex characters** → refuse naming the cause: git cannot fetch an abbreviated
        SHA; pass the full 40-character SHA. **This must be checked BEFORE the refname rules**,
        since a 7-hex string is a perfectly valid refname and would otherwise pass. (R5a.)
     5. refname-invalid forms: `..`, `~`, `^`, `:`, `?`, `*`, `[`, `\`, leading or trailing `/`,
        leading or trailing `.`, trailing `.lock`.
     6. otherwise accept, **verbatim and unnormalized** (it becomes provenance, per the
        `https://` precedent at `:113-115`).
  4. **Ordering guard:** a case that is *both* leading-dash and hex-looking asserts the dash
     refusal wins, mirroring `test_leading_dash_is_refused_even_when_it_also_looks_like_a_url`
     (`test_reproduce_remote_intake.py:89`).
- **Validation:** `uv run pytest tests/test_reproduce_rev_pin.py -q`
- **Commit:** `feat(reproduce): pure --rev validator refusing dashes, short SHAs, bad refnames`

### Phase 2 — Argv builders

- **Goal:** four fixed argv builders for the targeted fetch. Spec criterion 5.
- **Prerequisites:** none. Parallelizable with Phase 1.
- **Files:** `src/contig/runner.py` (add), `tests/test_reproduce_rev_pin.py` (extend).
- **Steps:**
  1. **RED:** assert each argv **exactly**, as `test_git_clone_argv_is_exact`
     (`test_reproduce_remote_intake.py:207`) does; plus a `--`-placement test per builder that
     accepts one, mirroring `:220`.
  2. Add beside `_git_clone_argv` (`runner.py:658-668`), each with a docstring saying why the
     shape is fixed:
     - `_git_init_argv() -> ["git", "init", "-q"]` (run **with `dest` as cwd**).
     - `_git_remote_add_argv(url) -> ["git", "remote", "add", "origin", "--", url]`
     - `_git_fetch_argv(rev) -> ["git", "fetch", "--depth", "1", "origin", "--", rev]`
     - `_git_checkout_argv() -> ["git", "checkout", "--detach", "FETCH_HEAD"]`
  3. Docstring note on `_git_fetch_argv`: `--depth 1` is kept for the same reason as the clone,
     and `--` is the second line of defence behind Phase 1's leading-dash refusal. **Verified**
     that `git remote add` and `git fetch` both accept `--`.
  4. Docstring note on `_git_checkout_argv`: `--detach` is explicit so the detached state is
     intentional rather than incidental; `rev-parse HEAD` is unaffected.
- **Validation:** `uv run pytest tests/test_reproduce_rev_pin.py -q`
- **Commit:** `feat(reproduce): fixed argv builders for the targeted rev fetch`

### Phase 3 — `fetch_repo` targeted-fetch branch

- **Goal:** fetch a named revision, validate the pin, clean up on every failure. Spec criteria
  6, 8, 9, 10.
- **Prerequisites:** Phases 1–2.
- **Files:** `src/contig/fetch.py`, `tests/test_reproduce_rev_pin.py`.
- **Steps:**
  1. **RED first, in this order:** happy path (SHA / tag / branch) asserting the **exact call
     sequence and each call's cwd**; then one failure test per git step; then the R4 mismatch;
     then no-litter-on-failure; then the R6 message.
  2. Extend the scripted fetcher: copy `_ScriptedFetcher`
     (`test_reproduce_remote_intake.py:254-268`) and adapt the filesystem side-effect — the
     **`git init` call (1)** creates nothing new (`dest` already exists), and the **checkout
     call (4)** is what materialises the worktree files. Keep the record-every-`(argv, cwd)`
     shape unchanged.
  3. Add a keyword-only `rev: str | None = None` to `fetch_repo` (`fetch.py:194`). **Default
     `None` keeps the slice-6 path byte-identical** — that is the back-compat guarantee.
  4. Keep everything before the clone **unchanged**: `dest.absolute()` (`:222`), the non-empty
     refusal (`:224-228`), `parent_created_here` (`:233`), the wipe/mkdir (`:237-238`), and
     `_cleanup` (`:240-241`). The branch happens only at the fetch step.
  5. When `rev is None` → the existing single `_git_clone_argv` call, untouched.
     When `rev` is set → four calls, **all with `dest` as cwd** (unlike the clone, which uses
     `dest.parent`; `git init` makes `dest` a repo, so every later step runs inside it). Each
     non-zero exit → `_cleanup()` + a refusal naming the failing step and git's output.
  6. **R6:** when the fetch step fails and its output matches the `not our ref` / `upload-pack`
     shape (case-insensitive substring, not a regex), append to the refusal that the remote may
     not enable `uploadpack.allowReachableSHA1InWant` and to try a tag or branch.
  7. Reuse the **unchanged** `rev-parse` + `_FULL_SHA_RE.fullmatch` tail (`:250-262`).
  8. **R4:** after validation, if `rev` is a 40-hex SHA and `commit.lower() != rev.lower()` →
     `_cleanup()` + refusal. Record lowercase. A tag/branch has nothing to compare against.
- **Validation:** `uv run pytest tests/test_reproduce_rev_pin.py tests/test_reproduce_remote_intake.py -q`
  — **the slice-6 file must pass untouched.**
- **Commit:** `feat(reproduce): targeted fetch checks out a caller-named revision`

## 6. Edge cases & error handling

| Case | Behaviour |
|---|---|
| `--rev` + local repo | refuse pre-run, nothing written (R2) |
| `--rev` + URL, no `--allow-fetch` | existing slice-6 refusal naming `--allow-fetch` |
| `--rev` leading `-` | refused first, unconditionally (R5) |
| `--rev` 7-to-39 hex | refused naming the cause: pass the full 40-char SHA (R5a) |
| requested SHA ≠ resolved SHA | refuse; never record a mismatched pin (R4) |
| remote refuses fetch-by-commit | refuse, naming `allowReachableSHA1InWant`, suggest tag/branch (R6, D1) |
| any of the 4 git steps fails | refuse surfacing git's output; cleanup scoped by `parent_created_here` (R8) |
| `git` binary absent | existing `default_fetcher` `(127, msg)` path, unchanged |
| no `--rev` | byte-identical to slice 6 (R1) |

**Fallbacks:** none by design. D1 chose honest refusal over a silent full clone.
